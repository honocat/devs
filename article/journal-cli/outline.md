# TypeScriptで作る「ジャーナルCLI」入門（Notionに保存する対話型CLI）

## この記事で扱うこと / 扱わないこと
- 扱うこと
  - TypeScript + Node.jsでCLIを作る全体像
  - 対話形式でジャーナルを入力し、Notionに保存する（Childrenブロックまで）
  - 実装の最小構成（骨組み）と、拡張しやすい設計の考え方
- 扱わないこと
  - Notionのデータベース設計の深掘り（最低限の前提のみ）
  - CLIフレームワーク選定の網羅（今回は「動く最小」を優先）

## 前提（想定読者）
- Node.js / npm（またはpnpm, yarn）を触ったことがある
- TypeScriptの基本がわかる（型・import/export）
- Notionを日常的に使っている（APIは初めてでもOK）

## なぜジャーナルをCLIにするのか
- ジャーナルは良習慣だが「続けるのが難しい」
- 紙に毎日書くのはハードルが高い
- CLIなら「コマンド1つ」で開始できる
- 対話形式にすると心理的ハードルが下がる
- 自分で自由にカスタマイズでき、続けやすい形に寄せられる

## この記事の位置づけ（前回記事との関係）
- `../fast-notion/`で「Notionにメモを保存する最小単位」を解説した（骨組みはほぼ同じ）
- 今回はその応用として「ジャーナル」を題材に、実装を段階的に解説する

## 完成イメージ（ゴール）
- 例：`keys-cli morning-journal`（`../../apps/keys-cli/`）を題材にする
- 実行すると対話形式で質問され、回答がNotionに保存される
- 最低限の要件
  - 今日のページ（または日付ベース）を作成/取得
  - 回答をChildrenブロックとして追記する

## 全体構成（アーキテクチャ）
- CLI層：コマンド定義、引数/オプション、実行フロー
- Prompt層：質問項目の定義、入力のバリデーション
- Domain層：ジャーナルの「質問と回答」のモデル（型）
- Notion層：APIクライアント、ページ作成、Children追記
- （応用）Local層：ローカルファイル保存、参照

## 事前準備
### Notion側の準備
- Integrationを作成しトークンを取得
- 保存先（ページ/DB）を用意し、Integrationを招待
- 必要なID（ページID/DB ID）を取得

### ローカル側の準備
- Node.jsバージョンの目安
- プロジェクト初期化（例：`npm init` / `tsconfig`）
- 環境変数でNotionトークン等を管理（`.env` + dotenv等）

## 実装ステップ1：CLIの骨組みを作る
- `bin`（エントリポイント）を用意する
- コマンド名とサブコマンド（例：`morning-journal`）を定義
- まずは「Hello」出力 → 実行できる状態を作る
- ここでのポイント
  - CLIの責務を薄く保つ（入出力とフロー制御に集中）
  - 後で拡張できるディレクトリ構成にする

## 実装ステップ2：対話形式の入力（Prompt）
- 質問項目（例）
  - 今日の気分 / 今日やること / ありがとう / 一言 など
- 入力の型を定義（`type JournalAnswer = ...`）
- バリデーション（空文字を許すか、最大長など）
- 出力の整形（Notionに渡しやすい形に変換）

## 実装ステップ3：Notionに保存する（ページ作成・Children追記）
### Notionクライアントの初期化
- トークン読み込み（環境変数）
- APIクライアント生成

### 保存先の決め方
- 日付ベース（例：YYYY-MM-DD）でページタイトルを作る
- 既存ページがあれば使い、なければ作成する

### Childrenブロックとして追記する
- 質問 → 見出し、回答 → 段落、のようにブロックを組み立てる
- 追記単位（1回の実行で1セクション追加、など）
- 失敗時の扱い（リトライ/メッセージ/ログ）

## 実装ステップ4：コマンドを「毎日使える」形に整える
- 実行速度・体感の改善（不要なI/Oを減らす）
- エラーメッセージをユーザー向けにする
- オプション例
  - `--date` で対象日を指定
  - `--dry-run` でNotionに送らず表示のみ
  - `--format` でブロック形式を切り替え

## 応用：ローカルに保存して参照する
- 目的
  - オフラインでも下書きできる
  - 過去ログを高速に検索できる
  - Notion障害時の保険になる
- 方針案
  - 実行ごとにJSON/Markdownでローカル保存
  - Notion送信に成功したら送信済みフラグを付ける
  - 参照コマンドを追加（例：`journal last`, `journal grep <word>`）

## カスタマイズの指針（続けやすさを優先）
- 質問数を減らす（最初は3問から）
- 時間帯別にコマンドを分ける（morning/evening）
- 1行ジャーナル（最小入力）モードを作る
- 自分のトリガーに合わせてエイリアス化する（例：`j`）

## まとめ
- CLI + 対話形式でジャーナルの継続ハードルを下げられる
- Notion保存（Children）まで作れると、他の用途にも応用できる
- 次にやると良いこと
  - テンプレ（質問セット）を設定ファイル化
  - 過去ログ参照を強化（ローカル + Notion検索）

## 付録（必要なら）
- トラブルシュート（トークン/権限/IDの間違い）
- サンプルのディレクトリ構成案
- コード全体のリンク（`../../apps/keys-cli/` の該当箇所）
